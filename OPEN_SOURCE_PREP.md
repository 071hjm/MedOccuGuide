# 开源发布准备说明

本文档面向当前项目的仓库维护者，用于在正式公开发布前完成代码、数据、模型、合规和文档层面的整理。

## 1. 目标

建议将本项目以“研究原型 / 工程原型”的方式开源，而不是直接按生产系统对外承诺。原因如下：

- 当前项目涉及职业健康相关业务，存在数据、规则、模型和建议文本的合规要求。
- 当前仓库仍包含开发机路径、本地数据库、日志和大文件资产。
- 当前模型依赖与资源组织适合本地研究环境，不完全适合第三方零配置复现。

## 2. 发布前必须确认的事项

### 2.1 数据合规

必须确认以下数据是否允许公开：

- `data.pkl`
- `demo_test_samples.json`
- `sample_cases_by_label.json`
- 其他由真实业务数据抽取、切片或脱敏得到的样本文件

建议：

- 若任何字段可反推个人或单位信息，不应直接公开。
- 若数据来自合作单位，应取得书面授权后再发布。
- 若无法公开，建议仅保留手工构造的最小示例输入。

### 2.2 模型与知识库合规

必须确认以下资产是否允许公开分发：

- `saved_models/`
- `processed_data/`
- `models/`
- `LLaMA-Factory/saves/...`
- `rag/` 中的标准文本、知识片段和参考范围内容

建议：

- 若模型训练依赖专有数据，优先公开训练代码与接口，不直接公开权重。
- 若知识库文本来自标准、指南或第三方材料，应补充来源说明并核查分发许可。

### 2.3 安全与默认配置

当前仓库包含本地演示账号：

- 用户名：`admin`
- 密码：`admin123`

建议：

- 公开前移除默认明文密码。
- 改为首次启动初始化管理员，或从环境变量读取管理员账号。

## 3. 当前仓库中建议移出或忽略的内容

以下内容一般不建议直接进入公开 Git 仓库：

- 原始数据与中间数据
- 本地训练产物
- 本地大模型权重和 LoRA 权重
- SQLite 数据库
- 下载日志、运行日志、安装日志
- Python 安装包和其他二进制安装文件
- 本地缓存目录和 offload 目录

## 4. 当前仓库中需要优先改造的代码点

### 4.1 路径配置

当前代码中仍存在开发机绝对路径，例如：

- `occupational_health_runtime.py`
- `platform_storage.py`
- `gradio_case_app.py`
- `start_platform.ps1`
- `启动平台.bat`

建议改造为：

- 基于项目根目录的相对路径
- 或 `.env` / YAML / JSON 配置文件
- 或命令行参数配置

### 4.2 资源可选化

建议将以下资源改造为“可选组件”：

- LightGBM 模型
- 处理产物
- RAG 知识库
- Qwen 基座
- LoRA 适配器

公开仓库时应允许项目至少以“无私有模型的演示模式”启动，并在界面或日志中明确说明缺失资源。

### 4.3 登录与数据库初始化

建议：

- 将默认账号密码从代码中移出
- 提供首次启动自动建库能力
- 数据库为空时自动初始化表结构
- 在 README 中说明默认数据库位置

## 5. 推荐的公开仓库结构

推荐保留：

- 核心代码
- 样式文件
- 启动脚本
- 规则示例
- 脱敏示例输入
- 说明文档

推荐移出：

- 真实数据
- 大模型权重
- 训练产物
- 本地数据库
- 本地日志

## 6. 推荐的开源仓库配套文件

建议至少补齐以下文件：

- `README.md`
- `.gitignore`
- `LICENSE`
- `CONTRIBUTING.md`
- `SECURITY.md`
- `CHANGELOG.md` 或 Release Notes

其中：

- `README.md` 面向使用者
- `OPEN_SOURCE_PREP.md` 面向维护者
- `CONTRIBUTING.md` 面向贡献者
- `SECURITY.md` 面向漏洞反馈

## 7. 建议的发布流程

### 第一步：清理仓库

- 删除不应公开的数据、模型、数据库和日志
- 确认 `.gitignore` 生效
- 检查是否仍存在硬编码本地路径和明文密码

### 第二步：补全文档

- 完成项目简介、架构说明、安装步骤、运行方式、常见问题
- 明确说明哪些资源不随仓库分发
- 明确说明当前版本是研究原型还是可交付平台

### 第三步：选择许可证

建议结合用途选择：

- MIT：最宽松
- Apache-2.0：更适合工程开源
- GPL-3.0：强调衍生开源

如涉及外部模型、知识库或第三方标准文本，还需同步核查其许可证兼容性。

### 第四步：整理版本信息

- 建议使用 `v0.1.0`、`v0.2.0` 这类语义化版本
- 首次发布建议标记为“Research Prototype”或“Demo Release”

### 第五步：建立 GitHub 仓库基础能力

建议启用：

- Issues
- Discussions
- Releases
- 基础 CI

## 8. 建议的最小公开版本定义

对于当前项目，较稳妥的首个公开版本建议具备以下特征：

- 代码仓库可以正常安装依赖
- 没有真实业务数据
- 没有默认公开分发的大模型权重
- 提供可运行的最小示例输入
- 提供清晰的资源准备说明
- 文档明确说明系统边界和适用范围

## 9. 建议的对外表述

对外发布时，建议采用以下定位：

“本项目为职业健康体检场景下的混合智能诊断与受控生成平台原型，集成专家规则、机器学习、可解释分析、RAG 检索和 LoRA 微调大模型，用于辅助主检结论分析与建议生成。仓库公开代码、文档和示例配置，不默认分发真实业务数据和本地模型权重。”

## 10. 最终发布前检查清单

- 已确认数据与模型的分发权限
- 已移除真实业务数据
- 已移除本地数据库与日志
- 已清理大文件和安装包
- 已确认无硬编码敏感信息
- 已补充 README
- 已补充 `.gitignore`
- 已选择许可证
- 已测试“新环境克隆后”的安装流程
- 已准备首个 Release 说明
